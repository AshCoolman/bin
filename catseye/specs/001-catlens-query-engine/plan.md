# Implementation Plan: CatLens Query Engine

**Branch**: `main` (trunk-based) | **Date**: 2026-04-11 | **Spec**: [spec.md](spec.md)
**Input**: `specs/001-catlens-query-engine/spec.md`

## Summary

Build CatLens as a local TypeScript query engine with one shared core (AST, selection
engine, render pipeline) consumed by two frontends: a terminal-first CLI and a
strategic MCP server. The core is structured as three packages in a monorepo.
The CLI is the daily-use frontend; MCP is the strategic AI-facing frontend.
Both compile user input to the same typed AST and traverse the same pipeline:
parse → discover → facts → evaluate → SelectionResult → render.

## Technical Context

**Language/Version**: TypeScript 5.x, strict mode
**Primary Dependencies**: `fast-glob` (discovery), `ignore` (gitignore), `simple-git`
(git facts), `commander` (CLI), `@modelcontextprotocol/sdk` (MCP), `picocolors`
(terminal output)
**Storage**: local files — lens store at `.catlens/` in repo root (one JSON file per lens)
**Testing**: vitest — unit tests for AST/engine/renderer; integration tests against
fixture repos under `test/fixtures/`
**Target Platform**: Node.js 22+, macOS/Linux/Windows
**Project Type**: monorepo CLI tool + MCP server + shared core library (npm workspaces)
**Dev runner**: `tsx` for development execution; `tsc` for type-checking and build
**Performance Goals**: "fast enough" — no numeric target; correctness and determinism
take precedence over raw speed
**Constraints**: fully local execution; no network access; no cloud dependencies
**Scale/Scope**: single developer to small team; repos up to ~100k files

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Check | Notes |
|-----------|-------|-------|
| I. Shared Core | ✅ | One `packages/core`; CLI and MCP import from it; core has no frontend imports |
| II. AST-First | ✅ | AST types defined in `core/query` before any frontend work |
| III. SelectionResult boundary | ✅ | Renderers receive `SelectionResult` only; no re-evaluation |
| IV. Terminal-First, MCP-Driven | ✅ | CLI is primary; no feature may be CLI-only by design |
| V. MCP as Strategic Frontend | ✅ | MCP package uses same core; no shell-out to CLI |
| VI. Explicit Query Semantics | ✅ | `and/or/not/unless` and nesting are first-class AST nodes |
| VII. Query Integrity | ✅ | Validator runs before every execution; formatter + linter in core |
| VIII. Deferred Scope | ✅ | No vector DB, no JS eval, no tree-sitter, no web UI in this plan |

No violations. Complexity Tracking section is not required.

**Post-Phase 1 re-check** (all pass):
- The `unless` form is a first-class AST node, not flattened to `and(not(...))` during parsing.
  Boolean semantics are fully preserved across DSL → AST → MCP JSON → DSL roundtrips.
- The lens file format stores the Query AST directly; no semantic information is lost on save/load.
- The MCP server uses no shell-out to the CLI binary; it imports `packages/core` directly.
- The hand-written recursive descent parser produces source spans, enabling precise validator errors.

### Source Code (repository root)

```text
packages/
├── core/                    # shared engine — no CLI or MCP imports
│   ├── src/
│   │   ├── query/           # AST types, parser, formatter, validator, linter
│   │   ├── engine/          # predicate evaluation, boolean composition, selection
│   │   ├── repo/            # file discovery, ignore handling, path normalization
│   │   ├── facts/           # file/section/diff/git fact collectors
│   │   ├── render/          # renderers: file-list, markdown, snippets, diff, JSON
│   │   └── lenses/          # save/load/list, fuzzy lookup integration
│   ├── package.json
│   └── tsconfig.json
│
├── cli/                     # terminal frontend — imports core only
│   ├── src/
│   │   └── index.ts         # commander entrypoint
│   ├── package.json
│   └── tsconfig.json
│
└── mcp/                     # MCP server frontend — imports core only
    ├── src/
    │   └── server.ts        # MCP tool definitions
    ├── package.json
    └── tsconfig.json

test/
└── fixtures/                # fixture repos for integration tests
    ├── ts-app/              # small TS/TSX app with varied file types
    ├── tagged-sections/     # open/close marker examples
    ├── git-history/         # authorship, age, diff, commit-message examples
    └── ignored-files/       # .gitignore + nested ignore examples

package.json                 # npm workspaces root
tsconfig.base.json           # shared TS config (strict mode)
```

**Structure Decision**: Three-package monorepo (npm workspaces). The `core` package is
the engine; `cli` and `mcp` are thin shells that wire core to their respective protocol.
This enforces the constitution's import direction rule: frontends import core, core
imports nothing from frontends.
